DataPilot · 2024

Plain-English to SQL, with an agent loop that knows when it's wrong.

Built so non-technical staff across Sales, Operations, and Finance could query business data without writing SQL or waiting on an analyst.

Role Sole builder
Stack Python · Claude API · Streamlit · SQL Server
Status In daily production use

The problem

Business users needed data but didn't write SQL. Traditional BI tools were too rigid; analyst bandwidth was finite. Generic text-to-SQL tools hallucinated column names the moment the schema grew past a handful of tables.

The approach

Two agents in a loop, collaborating on every query:

  • Expert agent receives the user's question and schema documentation, generates a SQL query.
  • Tester agent receives the generated SQL and validates it against the schema before execution — catching missing joins, non-existent columns, wrong aggregations.
  • If the Tester fails the SQL, its feedback becomes context for the Expert's next attempt. Up to three retries.

The loop terminates when the Tester passes and the SQL executes cleanly, or returns a clear failure message with the full reasoning trace.

Why two agents, not one

A single agent asked to "generate and verify" conflates two different modes of thinking. Separation has three concrete benefits: the Tester can be stricter without compromising the Expert's generative freedom; failures are attributable (you know whether the Expert is hallucinating or the Tester is over-rejecting); and you can swap in a smaller, cheaper model for the Tester once the pattern is validated.

What worked

The reasoning trace panel turned out to be the feature staff cared about most. When SQL failed, they could read the Tester's feedback and understand why. When it succeeded, they could see the SQL and start learning the patterns. It turned DataPilot from a black box into a teaching tool.

What I'd change

The current version stuffs the full schema into every prompt. That works at eight tables and breaks at fifty. The v2 upgrade I'm building now is RAG over indexed schema documentation — retrieve only the slices relevant to the question, cut token usage, improve accuracy on ambiguous queries.
